# 第 29 章 · 低功耗设计：先测量，再进入睡眠（SPL版）

> **本章产出**：理解 Sleep、Stop、Standby 的区别；能为 STM32F1 设计一次可恢复的 Stop 模式唤醒流程。
>
> **前置知识**：第 5 章时钟、第 6 章中断，以及第 28 章调试方法。
>
> **用在哪**：电池供电的传感器节点、低频采样设备、待机控制器。

---

## 29.1 低功耗不是调用一个函数就结束

真正的功耗由整块系统决定：

~~~text
MCU 时钟 + 外设时钟 + 传感器 + WiFi/BLE 模块 + LED + 电源转换器
~~~

如果 WiFi 模块持续联网、OLED 常亮、LED 常亮，即使 MCU 进入低功耗，整机续航也不会明显改善。第一步永远是测量基线电流，再逐项关闭模块。

## 29.2 三种常见模式

| 模式 | CPU | 外设/时钟 | 唤醒后 | 典型用途 |
|---|---|---|---|---|
| Sleep | CPU 停止 | 多数时钟继续 | 从下一条指令继续 | 短时间等待中断 |
| Stop | 主时钟停止 | 大部分高速时钟关闭 | 需恢复时钟与外设 | 周期采样 |
| Standby | 接近复位状态 | 保留极少部分 | 类似复位启动 | 长时间待机 |

不同芯片和板子细节不同，任何低功耗设计都要回到对应参考手册验证。

## 29.3 先选择唤醒源

进入 Stop 前必须知道“谁来叫醒它”。常见选择：

- RTC Alarm：每 10 分钟采样一次；
- 外部按键或门磁：通过 EXTI 唤醒；
- WKUP 引脚：从 Standby 唤醒；
- 某些外设事件：取决于芯片和时钟配置。

WiFi 的 UART RX 并不自动等于可靠唤醒源。确认模块、引脚、EXTI 和低功耗模式组合后再设计。

## 29.4 一个明确的 Stop 恢复接口

第 5 章已经定义了 72 MHz 的 SystemClock_Config。低功耗章节应复用这个明确的时钟初始化函数，而不是再引入一个没有定义的新名字。

~~~c
void SystemClock_Config(void);  /* 第 5 章的 72MHz 时钟配置 */
void Periph_ResumeAfterStop(void); /* 重新配置依赖时钟的外设 */

void EnterStopMode(void)
{
    RCC_APB1PeriphClockCmd(RCC_APB1Periph_PWR, ENABLE);

    PrepareWakeupSource();  /* 例如 RTC 或 EXTI */
    FlushPendingLog();

    PWR_EnterSTOPMode(PWR_Regulator_LowPower, PWR_STOPEntry_WFI);

    /* 从中断唤醒后，系统通常回到低速时钟 */
    SystemClock_Config();
    Periph_ResumeAfterStop();
}
~~~

不要假设所有外设状态都会自动恢复。UART 波特率、I2C、SPI、SysTick 和外部模块的状态都应在测试板上逐一确认。

## 29.5 传感器节点的工作周期

~~~text
唤醒
  → 恢复时钟
  → 供电并初始化传感器
  → 采样
  → 本地记录 / 条件上报
  → 关闭传感器和不必要外设
  → 设置下一次唤醒
  → Stop
~~~

如果网络上报耗时很长，通常不要每次采样都联网。可以多次本地缓存，再批量连接发送，以换取更长续航。

## 29.6 测试方法

| 测试 | 目的 |
|---|---|
| 运行态电流 | 找到最大的持续耗电源 |
| Stop 电流 | 验证 MCU 和板级外围是否真的降下去 |
| 唤醒成功率 | 连续数百次休眠/唤醒不丢功能 |
| 时钟恢复 | 唤醒后 UART 波特率、I2C、SysTick 正常 |
| 数据完整性 | 休眠前后 seq 和日志不乱 |

万用表适合看平均电流，示波器或电流分析仪更适合看短暂峰值。记录“电压、模式、外设状态、测得电流”，不要只记录一个漂亮数字。

## 29.7 常见误区

- 只调用 WFI，却没有关闭任何外设；
- 进入 Stop 后忘记恢复时钟；
- 用延时等待外设，却让 CPU 忙等；
- 忽略开发板上的稳压器、LED 和 USB-TTL 的静态功耗；
- 没有准备唤醒源，设备“睡死”。

## 29.8 开发板级功耗陷阱

测量低功耗时，开发板上的耗电往往比 MCU 更显眼：

| 部件 | 可能造成的误判 |
|---|---|
| 电源指示 LED | MCU 已睡眠但 LED 仍持续耗电 |
| USB-TTL | 串口芯片和 USB 供电掩盖真实功耗 |
| 板载 LDO | 静态电流可能高于预期 |
| WiFi/BLE 模块 | 发射瞬间峰值远高于待机电流 |
| 传感器/OLED | 没有断电或休眠时持续消耗 |

因此应分别测“整块开发板”和“目标设备最小硬件”的电流。先记录基线，再关闭一个模块测一次；不要只给出一个没有条件说明的电流数字。

## 29.9 本章练习

1. 让 LED 每 10 秒闪一次，其余时间进入 Sleep；
2. 改为 RTC 或 EXTI 唤醒的 Stop 实验；
3. 对比三种状态下的平均电流；
4. 写一张表，列出每个外设进入休眠前的关闭与恢复动作。

## 29.10 Stop 模式的恢复契约与测量实验

进入 Stop 前和唤醒后是同一套接口的两端。对 F1 而言，Stop 后需要重新建立系统时钟，并按项目依赖恢复外设；不要只调用睡眠函数后就假设 UART、I2C、WiFi 都还在原状态。

~~~c
void Power_EnterStop(void)
{
    /* 先停止或保存本项目依赖的外设状态：UART 发送完成、I2C 空闲、
       DMA 停止、外部模块进入可恢复状态。 */
    PWR_EnterSTOPMode(PWR_Regulator_LowPower, PWR_STOPEntry_WFI);

    /* 从唤醒点继续：重新配置系统时钟，再恢复必要外设。 */
    SystemClock_Config();       /* 复用第 5 章已经验证的时钟路径 */
    Board_PeripheralsResume();  /* 由项目明确实现，不应是空名字 */
}
~~~

上例不是可复制到所有工程的完整函数；它强调恢复顺序。你的 `Board_PeripheralsResume()` 应列出具体动作：时钟、UART 波特率、I2C 状态、传感器电源、无线模块和 RTOS tick 相关策略。

### 电流测量实验

| 步骤 | 做什么 | 记录什么 |
|---|---|---|
| 基线 | 正常运行、无线关闭 | 整板平均/峰值电流、供电方式 |
| 关闭外设 | 每次只关一个模块 | 电流变化与功能影响 |
| Sleep | 保持时钟/外设策略明确 | 唤醒延迟和电流 |
| Stop | 多次 EXTI/RTC 唤醒 | 时钟恢复、UART 是否仍正确 |
| 长循环 | 至少数百次睡醒 | 错误、丢帧、重连和电流趋势 |

万用表适合平均值；短峰值和唤醒瞬态通常需要示波器或电流分析工具。开发板上的 LED、电源 LED、USB-TTL、LDO 与 WiFi 模块常常比 MCU 本体更耗电，因此必须分别测整板和最小目标硬件。

练习：为每个外设做一张“进入前关闭动作 / 醒来后恢复动作 / 验证证据”表，然后让系统连续唤醒 500 次。

## 29.11 用功耗预算设计采样周期

低功耗的目标不是“某一瞬间电流小”，而是周期平均电流和唤醒任务是否符合需求：

~~~text
I_avg = (I_run × t_run + I_sleep × t_sleep + I_radio × t_radio) / T
T     = t_run + t_sleep + t_radio
~~~

先用测量值填表，而不是从宣传页抄一个典型电流：

| 阶段 | 持续时间 | 实测电流 | 贡献 |
|---|---:|---:|---:|
| 唤醒 + 时钟恢复 | … ms | … mA | … |
| 传感器预热/采样 | … ms | … mA | … |
| 无线连接/发送 | … ms | … mA | … |
| Stop/Sleep | … ms | … µA/mA | … |
| 一个周期 | … s | 平均 | `I_avg` |

### 低功耗前后的硬件状态表

| 对象 | 入睡前 | 醒来后 | 验证 |
|---|---|---|---|
| 系统时钟 | 记录当前配置 | 重新配置 PLL/总线时钟 | UART 波特率/定时器周期 |
| UART | 等待 TX 完成或停止发送 | 恢复波特率/接收路径 | 固定心跳文本 |
| I2C/SPI | 总线空闲、CS 安全 | 重新初始化或探测从机 | ACK/设备 ID |
| 传感器 | 关电或低功耗命令 | 等待稳定时间 | 合理读数 |
| WiFi/BLE | 按模块手册休眠/断开 | 明确重连策略 | 连接/错误计数 |
| GPIO | 避免浮空与无意义输出 | 恢复复用配置 | 万用表/逻辑分析仪 |

练习：先让 WiFi 模块完全不参与，只优化 MCU + 单传感器周期；再单独加回无线，比较它对平均电流的影响。这样不会把模块待机、电源 LED 或 USB-TTL 的耗电误归因给 MCU。

## 29.12 F103 Stop 的精确恢复边界：时钟、RTOS 和测量夹具

对 STM32F103，Stop 唤醒后代码从 `WFI` 之后继续，但不能把“从函数返回”理解成“72MHz 的原时钟树已自动回来”。按 RM0008 的 PWR/RCC 说明，进入 Stop 后由软件重新建立高速时钟路径；本书的 `SystemClock_Config()` 因此必须是已在第 5 章独立验证过、能重新启用 HSE/PLL 并切回目标系统时钟的函数。不同晶振、板载电路和项目时钟树仍要逐项实测。

把一次 Stop 实验的契约写成：

~~~text
选择已验证的唤醒源（RTC Alarm 或 EXTI）
  → 清理/记录该唤醒源的 pending 标志
  → 停止或完成 UART/DMA/总线事务
  → 进入 STOP + WFI
  → 唤醒 ISR 只留下事件
  → SystemClock_Config() 恢复 HSE/PLL/总线时钟
  → 恢复 SysTick、外设和任务依赖
  → 用 UART 波特率、定时器周期和一次传感器读数验证
~~~

### FreeRTOS 不能由任意任务“随手 Stop”

若 FreeRTOS 的 SysTick 仍按普通 tick 工作，进入 Stop 后 tick 不会像平时一样推进。不要让一个业务任务直接调用 `PWR_EnterSTOPMode`，同时假定所有延时和超时仍正确。选择其一并在项目中写清楚：

- 只在没有调度器的最小低功耗实验中验证 Stop；
- 使用已理解且已测试的 tickless idle/电源管理路径，并补偿经过的时间；
- 先使用 Sleep，等任务、唤醒源和计时语义清楚后再引入 Stop。

本书不把上面的教学骨架宣称为一个完整的 FreeRTOS tickless port。

### 正确的测量夹具比一个漂亮的 µA 数字重要

低功耗测量应记录供电路径。调试时 USB、ST-Link、USB-TTL、板载 LED 和 LDO 都会影响结果；要测 MCU/目标板基线时，使用受控外部供电并明确哪些调试设备仍连接。电流表串联会引入负担电压，可能让 WiFi 发射或唤醒瞬态在“测量时”异常；因此平均电流、短峰值和整板/最小目标板应分别记录。

最后把电池设计也写成假设而非结论：

~~~text
理论工作时间（小时）≈ 标称容量(mAh) × 可用比例 / 实测 I_avg(mA)
~~~

可用比例要考虑温度、老化、LDO 截止、电池保护和无线峰值；没有这些条件，就不要从一次 Stop 电流推导“可用几个月”。

## 29.13 本章要点

- 低功耗是系统工程，先测量再优化；
- Sleep、Stop、Standby 的唤醒成本不同；
- Stop 后必须明确恢复时钟和依赖外设；
- 唤醒源与模式必须作为同一个设计问题；
- 能连续验证数百次的休眠/唤醒，才算可靠。
